  <action>
  Create docs/s3-lifecycle.md as the authoritative S3 retention runbook.

  The document must cover all five of the following sections (in this order):

  1. Overview
     - What the lifecycle policy does: expires backups/postgres/ objects after 30 days, aborts
       incomplete multipart uploads after 7 days.
     - Requirement IDs satisfied: S3-01 (policy stored + script-applied), S3-02 (AbortIncompleteMultipartUpload),
       S3-03 (empirical proof).
     - Operator-gate note: applying the real policy to the production backup bucket is a consequential,
       retention-affecting action. Run the empirical probe first. Confirm S3-03 evidence before running apply.

  2. Lifecycle Policy File
     - Location: config/s3/backups-lifecycle.json
     - Rule 1 (ID expire-postgres-backups): Filter.Prefix=backups/postgres/, Expiration.Days=30.
       Justification: 30 daily backups retained; conservative window; operator may raise Days before
       first apply without any code change.
     - Rule 2 (ID abort-incomplete-multipart): AbortIncompleteMultipartUpload.DaysAfterInitiation=7.
       Justification: standard AWS recommendation; bucket-wide; only affects in-progress uploads.
     - Note: NO secrets in this file; safe to commit.

  3. Apply Procedure (S3-01)
     - Pre-requisite: cluster accessible (WireGuard up), KUBECONFIG configured.
     - Pre-requisite: run the empirical probe (Section 4) and record S3-03 evidence before proceeding.
     - Step-by-step:
       a. Review config/s3/backups-lifecycle.json to confirm the Expiration.Days value is acceptable.
       b. Run: scripts/apply-s3-lifecycle.sh
          (The script creates a k8s Job that applies the policy from inside the cluster using
          server-2-runtime S3 credentials. It prints WARN if an existing config is found.)
       c. Confirm the Job completes successfully and logs "lifecycle configuration applied successfully".
       d. Verify with: kubectl logs job/<job-name> -n solid-stats-staging
     - Idempotent: re-running apply-s3-lifecycle.sh is safe; it will WARN if a config exists but
       still apply (operator reviews the WARN and confirms intent).

  4. Empirical Proof Procedure (S3-03)
     - Why: Timeweb S3 lifecycle API parity is MEDIUM confidence. Prove before trusting.
     - Pre-requisite: same cluster access as above.
     - Step-by-step:
       a. Apply the probe Job: kubectl apply -f k8s/staging/s3-lifecycle/80-s3-lifecycle-probe-job.yaml
       b. Wait: kubectl wait --for=condition=complete job/s3-lifecycle-probe -n solid-stats-staging --timeout=120s
       c. Collect logs: kubectl logs job/s3-lifecycle-probe -n solid-stats-staging
       d. Confirm the API support result ("API implemented" vs "NOT implemented").
       e. Observe x-amz-expiration in head-object output. Note: x-amz-expiration will only be present
          if a lifecycle rule already targets the s3-lifecycle-probe/ prefix or a bucket-wide rule exists.
          After running apply-s3-lifecycle.sh, add a temporary rule for s3-lifecycle-probe/ and re-run
          the probe to see the expiry header.
       f. Confirm cleanup: "probe cleanup complete" in logs.
       g. Delete the Job: kubectl delete job s3-lifecycle-probe -n solid-stats-staging
       h. Record evidence in the Evidence section below.

  5. Evidence (S3-03)
     - Provide a table or placeholder section with these fields, to be filled in by the operator:
       | Field | Value |
       |-------|-------|
       | Date of probe run | |
       | Cluster endpoint | solid-stats-staging |
       | API support result | (API implemented / NOT implemented) |
       | Existing lifecycle config at probe time | (none / found — describe) |
       | x-amz-expiration present | (yes / no) |
       | Expiry date observed | |
       | Cleanup confirmed | (yes) |
       | Operator | |
     - Add a note: "This section must be filled before the apply procedure (Section 3) is run
       for the first time. A blank evidence table means the policy has NOT been proven."

  6. Async Expiry Caveat
     - S3 lifecycle expiration is asynchronous: objects are not deleted at midnight on day 30.
       AWS/Timeweb typically processes expiry within 24 hours after the expiry date (Days count
       from object creation date, not upload date). Do not rely on exact-to-the-hour deletion.
     - Monitoring: check S3 object counts via aws s3 ls s3://$S3_BUCKET/backups/postgres/ --recursive
       periodically after the policy is applied to confirm old objects are being expired.
     - The x-amz-expiration header in head-object is the best available signal that the rule
       is recognized; actual deletion is confirmed by observing object count decrease over time.

  Style: match docs/backup-restore.md (markdown, operator-facing, direct prose, numbered steps).
  No emojis.
