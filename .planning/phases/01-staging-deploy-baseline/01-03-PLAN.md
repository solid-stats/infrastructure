<objective>
Make the deploy path and operator verification output explicit enough that an
operator can apply the full staging baseline from this repository and see which
resources are expected to be healthy.
</objective>

<threat_model>
- T-01-07: Deploy succeeds partially but operator misses unhealthy resources.
  Mitigation: deploy script prints rollout checks and expected resource list.
- T-01-08: Secret manifests remain on the remote host. Mitigation: keep remote
  cleanup behavior and avoid printing rendered secrets.
</threat_model>

<action>
Update `scripts/deploy-staging.sh` so after applying manifests it prints a
concise verification table or clearly separated commands for:
- `kubectl -n "$namespace" rollout status statefulset/postgres --timeout=300s`
- `kubectl -n "$namespace" rollout status statefulset/rabbitmq --timeout=300s`
- `kubectl -n "$namespace" rollout status deployment/server-2 --timeout=300s`
- `kubectl -n "$namespace" rollout status deployment/replay-parser-2 --timeout=300s`
- `kubectl -n "$namespace" get service postgres rabbitmq server-2 -o wide`
- `kubectl -n "$namespace" get cronjob replays-fetcher postgres-backup -o wide`
Do not force-run either CronJob in Phase 1.
</action>

- `grep -q "rollout status deployment/replay-parser-2" scripts/deploy-staging.sh`
- `grep -q "get service postgres rabbitmq server-2" scripts/deploy-staging.sh`
- `grep -q "get cronjob replays-fetcher postgres-backup" scripts/deploy-staging.sh`
</acceptance_criteria>
