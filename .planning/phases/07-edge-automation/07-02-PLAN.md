<objective>
Create the certbot failure surfacing drop-in (config/systemd/certbot.service.d/onfailure.conf),
the failure handler unit (config/systemd/certbot-renew-failure.service), and the deploy hook
script (config/systemd/certbot-deploy-hook.sh).

The stock certbot.timer is already running twice daily on the live host (per D-4) — we do NOT
create a parallel timer or service. The repo value-add is:
  - A systemd drop-in that wires OnFailure= onto the stock certbot.service, routing failures
    to our journald logger (EDGE-03, D-5).
  - A deploy hook in /etc/letsencrypt/renewal-hooks/deploy/ that gates nginx reload on
    nginx -t (EDGE-02, D-4).

Purpose: Adding these two pieces turns silent renewal failures into visible journald entries
and guards nginx reload against broken configs — without interfering with the stock renewal
schedule already in place.

Output: config/systemd/certbot.service.d/onfailure.conf, config/systemd/certbot-renew-failure.service,
config/systemd/certbot-deploy-hook.sh (per EDGE-02, EDGE-03).
</objective>

  <action>
    Create two files. Both are installed to /etc/systemd/system/ by bootstrap-edge.sh (Plan 03).

    FILE 1 — config/systemd/certbot.service.d/onfailure.conf:

    This is a systemd drop-in for the STOCK certbot.service unit (the one managed by apt
    certbot on Ubuntu 24.04 noble). A drop-in is a file in /etc/systemd/system/certbot.service.d/
    that overrides or extends fields in the stock unit without replacing it (per D-5).

    The drop-in adds ONLY the OnFailure= directive. It must NOT replace ExecStart, the timer
    schedule, or any other field — the stock certbot.service already handles renewal.

    Content:
    # Source: config/systemd/certbot.service.d/onfailure.conf
    # Drop-in for the stock certbot.service (Ubuntu 24.04 apt-managed).
    # Adds OnFailure= wiring to surface renewal failures to journald.
    # DO NOT add ExecStart or other overrides — this drop-in extends, not replaces.
    # Installed to: /etc/systemd/system/certbot.service.d/onfailure.conf
    # Applied by: scripts/bootstrap-edge.sh (operator-run)
    [Unit]
    OnFailure=certbot-renew-failure.service

    FILE 2 — config/systemd/certbot-renew-failure.service:

    This is the failure handler unit activated when certbot.service fails. It writes a
    crit-priority syslog entry via logger so the operator can check journalctl.

    Content:
    # Source: config/systemd/certbot-renew-failure.service
    # Activated by OnFailure= in certbot.service.d/onfailure.conf.
    # Logs a user.crit entry to journald when certbot renewal fails.
    # Installed to: /etc/systemd/system/certbot-renew-failure.service
    # Applied by: scripts/bootstrap-edge.sh (operator-run)
    [Unit]
    Description=Log Certbot Renewal Failure to journald
    After=certbot.service

    [Service]
    Type=oneshot
    ExecStart=/bin/bash -c 'logger -t certbot-alert -p user.crit "Certbot renewal FAILED. Inspect: journalctl -u certbot.service -n 20"'

    [Install]
    WantedBy=multi-user.target

    Notes:
    - logger -t certbot-alert -p user.crit writes a crit-priority syslog entry with tag "certbot-alert".
    - Operator reads: journalctl -t certbot-alert --since today (or journalctl -p crit --since today).
    - No external monitoring required; no email/webhook; journald is sufficient for solo operator (per D-5).
    - The failure message directs to the actual renewal logs (certbot.service, not certbot-renew.service).
    - Drop-in dir name certbot.service.d corresponds exactly to the unit name certbot.service (systemd convention).
  </action>

  <name>Task 2: certbot deploy-hook — config/systemd/certbot-deploy-hook.sh (EDGE-02, D-4)</name>
  <files>config/systemd/certbot-deploy-hook.sh</files>
  <action>
    Create config/systemd/certbot-deploy-hook.sh. This script is installed by bootstrap-edge.sh
    to /etc/letsencrypt/renewal-hooks/deploy/reload-nginx.sh. The stock certbot.timer calls all
    scripts in the deploy/ directory after each successful renewal (per D-4).

    Header:
    #!/usr/bin/env bash
    set -euo pipefail

    Header comment block:
    # certbot-deploy-hook.sh — post-renewal nginx reload gate
    # Source: config/systemd/certbot-deploy-hook.sh — managed by infrastructure repo
    # Installed by: scripts/bootstrap-edge.sh to /etc/letsencrypt/renewal-hooks/deploy/reload-nginx.sh
    # Called by: certbot (stock certbot.timer) after each successful certificate renewal.
    # certbot sets $RENEWED_DOMAINS before calling this hook.

    Implementation:

    TIMESTAMP=$(date +'%Y-%m-%d %H:%M:%S')
    RENEWED="${RENEWED_DOMAINS:-unknown}"
    echo "[$TIMESTAMP] [certbot-deploy-hook] Renewal hook triggered for: $RENEWED"

    Section 1 — nginx config validation (fail-closed gate, per EDGE-02):
    echo "[$TIMESTAMP] [certbot-deploy-hook] Validating nginx configuration..."
    if ! nginx -t 2>&1; then
      echo "[$TIMESTAMP] [certbot-deploy-hook] FATAL: nginx -t failed — refusing reload" >&2
      echo "[$TIMESTAMP] [certbot-deploy-hook] New certificate installed but nginx NOT reloaded" >&2
      echo "[$TIMESTAMP] [certbot-deploy-hook] Fix nginx config and run: systemctl reload nginx" >&2
      exit 1
    fi

    Section 2 — nginx reload:
    echo "[$TIMESTAMP] [certbot-deploy-hook] Reloading nginx..."
    systemctl reload nginx

    Section 3 — success log:
    TIMESTAMP=$(date +'%Y-%m-%d %H:%M:%S')
    echo "[$TIMESTAMP] [certbot-deploy-hook] Success — nginx reloaded with renewed certificate(s)"

    IMPORTANT: nginx -t MUST appear before systemctl reload nginx — the ordering is the
    safety gate. If nginx -t exits non-zero, script exits 1 and the stock certbot marks
    the deploy hook as failed (visible in journalctl -u certbot.service). This failure
    propagates through the OnFailure= drop-in from Task 1 to log a user.crit alert.

    The repo file does not need execute permission — bootstrap-edge.sh runs chmod +x on
    the host copy at /etc/letsencrypt/renewal-hooks/deploy/reload-nginx.sh.
  </action>

  </verify>
