<objective>
Create a deterministic validation entrypoint that CI runs before deploy. The
validator must check script syntax, manifest shape, workload safety invariants,
and rendered Secret structure without printing real secret values.
</objective>

<threat_model>
- T-01-01: Secret values leak through CI logs. Mitigation: validation uses dummy
  environment values and asserts structure only.
- T-01-02: Broken manifests reach staging. Mitigation: parse every YAML
  document and fail before deploy.
- T-01-03: GHCR pull Secret is malformed. Mitigation: decode
  `.dockerconfigjson` and assert `auths.ghcr.io.username`, `password`, and
  `auth` exist.
</threat_model>

<action>
Add `scripts/validate-staging.py` using only the Python standard library. It
must:
- run `py_compile` on `scripts/render-staging-secrets.py`
- run `bash -n` on `scripts/deploy-staging.sh` and
  `scripts/backup-postgres-now.sh`
- verify the expected manifest files exist
- parse all YAML documents through a minimal safe parser for the repository's
  simple Kubernetes YAML shape, or invoke `kubectl apply --dry-run=client -f -`
  when `kubectl` is available and otherwise perform structural checks
- fail when a Kubernetes document lacks `apiVersion` or `kind`
- fail when workload pod specs rely on the default ServiceAccount without an
  explicit documented exception
- fail when containers lack resource requests or limits
- fail when long-running Deployment/StatefulSet containers lack both readiness
  and liveness probes
- fail unless either NetworkPolicy manifests exist or `docs/staging.md`
  contains `NetworkPolicy exception`
- run `scripts/render-staging-secrets.py` with deterministic dummy environment
  values and assert required Secret names, types, keys, and GHCR Docker auth
  structure
- print only pass/fail messages and object names, never dummy secret values
</action>
