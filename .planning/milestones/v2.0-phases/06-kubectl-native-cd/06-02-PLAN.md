    Title: "ServiceAccount Token and WireGuard Key Rotation"

    Intro paragraph: Explains the purpose (long-lived SA token requires periodic rotation for security), cadence (at minimum quarterly), who owns it (the operator — the person who performed the initial bootstrap). States that SA token rotation and WireGuard key rotation must be coordinated in the same maintenance window. Links to docs/operator-bootstrap.md for context on initial setup. GitHub environment secrets (K8S_TOKEN, WG_PRIVATE_KEY, WG_PEER_PUBLIC_KEY) must be updated BEFORE rotating the VPS side so no deploy fails mid-rotation.

    ## Overview
    - Owner: Operator (the person who runs the bootstrap runbook)
    - Cadence: At minimum quarterly; rotate immediately if a token or key is suspected compromised
    - Scope: SA token (K8S_TOKEN) and WireGuard key pair (WG_PRIVATE_KEY + WG_PEER_PUBLIC_KEY) — both in the same window
    - Window coordination: Update GitHub secrets first, then rotate on the VPS

    ## Step 1: Rotate the ServiceAccount token

    Sub-steps:
    1a. Delete the old token Secret: `kubectl delete secret ci-deployer-token -n solid-stats-staging`
    1b. Re-apply the Secret manifest: `kubectl apply -f k8s/staging/01-ci-rbac.yaml` (the Secret definition is in this file; deleting and re-applying forces the control plane to generate a new token)
    1c. Wait for the new token to be populated (a few seconds): `kubectl get secret ci-deployer-token -n solid-stats-staging -o jsonpath='{.data.token}' | base64 -d | wc -c` — should return non-zero length
    1d. Extract new token: `kubectl get secret ci-deployer-token -n solid-stats-staging -o jsonpath='{.data.token}' | base64 -d`
    1e. Update GitHub environment secret K8S_TOKEN (Settings → Environments → staging → K8S_TOKEN) with the new token value

    ## Step 2: Rotate the WireGuard key pair

    Sub-steps:
    2a. Generate a new WireGuard private key for the CI peer: `wg genkey` — store securely; do NOT commit
    2b. Derive the new public key: `echo "<private-key>" | wg pubkey`
    2c. Update GitHub environment secrets:
        - WG_PRIVATE_KEY → new private key
        - WG_PEER_PUBLIC_KEY → new public key (this is the CI peer's public key as seen by the VPS)
    2d. Update the VPS WireGuard config: Replace the existing CI peer's `PublicKey` in the VPS wg0 config with the new public key, then `sudo wg syncconf wg0 <(sudo wg-quick strip wg0)` or `sudo systemctl restart wg-quick@wg0`
    2e. Verify handshake: A subsequent CI deploy run must show a completed handshake (or manually test with `sudo wg show wg0 latest-handshakes` from VPS showing a recent timestamp for the CI peer)

    ## Step 3: Verify both rotations

    3a. Trigger the deploy workflow on a feature branch (or workflow_dispatch) — the validate + dry-run jobs must complete without authentication errors
    3b. Confirm `kubectl auth whoami` in the dry-run job log shows the ci-deployer ServiceAccount, not system:anonymous
    3c. Confirm no "Unauthorized" errors in the deploy job log

    ## Troubleshooting

    - "Unauthorized" after token rotation: Check that K8S_TOKEN in GitHub secrets contains the new token value (not the old one); re-extract from Secret if needed
    - "WireGuard handshake did not complete": VPS peer config may not have been updated yet, or the new public key in WG_PEER_PUBLIC_KEY does not match the new private key in WG_PRIVATE_KEY — regenerate the pair and redo Step 2
    - Rotation fails mid-window: The old GitHub secret is still valid; revert K8S_TOKEN / WG_PRIVATE_KEY to the old values while diagnosing

    ## Related Documents

    - docs/operator-bootstrap.md — initial setup of SA and RBAC (first rotation follows same Secret deletion pattern)
