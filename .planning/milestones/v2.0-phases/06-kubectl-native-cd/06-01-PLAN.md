    - labels: app.kubernetes.io/name: ci-deployer, app.kubernetes.io/part-of: solid-stats

    - rules covering exactly what kubectl apply and kubectl rollout status require:
      - apiGroups: ["apps"], resources: ["deployments", "statefulsets"], verbs: ["get", "list", "watch", "create", "update", "patch"]
      - apiGroups: ["batch"], resources: ["cronjobs"], verbs: ["get", "list", "watch", "create", "update", "patch"]
      - apiGroups: [""], resources: ["pods"], verbs: ["get", "list", "watch"]
      - apiGroups: [""], resources: ["services", "configmaps", "secrets", "persistentvolumeclaims"], verbs: ["get", "list", "create", "update", "patch"]
      - apiGroups: [""], resources: ["serviceaccounts"], verbs: ["get", "list", "create", "update", "patch"]
    - Do NOT add DaemonSets — no DaemonSets exist in staging; do not grant more than needed
    - Do NOT add cluster-scoped resources (namespaces, nodes, clusterroles, etc.)

    Document 4 — RoleBinding:
    - name: ci-deployer, namespace: solid-stats-staging
    - roleRef: kind: Role, name: ci-deployer, apiGroup: rbac.authorization.k8s.io
    - subjects: kind: ServiceAccount, name: ci-deployer, namespace: solid-stats-staging

    NOTE: This file is operator-applied via `kubectl apply -f k8s/staging/01-ci-rbac.yaml` — it is NOT applied by CI.     Create docs/operator-bootstrap.md following the docs/backup-restore.md runbook format. This is a one-time operator action; CI never runs these steps. Structure:

    Title: "Operator Bootstrap: Namespace, CI RBAC, and k3s Certificate SAN"

    Intro paragraph: Explains this is a one-time setup by the operator. States that CI (GitHub Actions) never creates the namespace (cluster-scoped) and never applies 01-ci-rbac.yaml. Links to sa-token-rotation.md for rotation cadence.

    ## Prerequisites
    - kubectl access to the k3s cluster with admin privileges
    - Git clone of this repository
    - Access to the GitHub repository environment secrets (Settings → Environments → staging)

    ## Step 1: Create the namespace
    - `kubectl create namespace solid-stats-staging` (or verify it already exists)
    - State: CI never creates the namespace; if a deploy fails with "namespace not found", the operator must re-run this step.

    ## Step 2: Apply RBAC bootstrap manifest
    - `kubectl apply -f k8s/staging/01-ci-rbac.yaml`
    - Explains this creates the ServiceAccount, token Secret, Role, and RoleBinding.
    - k8s control plane auto-populates the token in the Secret within a few seconds.

    ## Step 3: Verify RBAC
    - Command: `kubectl auth can-i --list --as=system:serviceaccount:solid-stats-staging:ci-deployer -n solid-stats-staging`
    - Expected output guidance: should list get/list/watch/create/update/patch on deployments, statefulsets, cronjobs, configmaps, secrets, services, persistentvolumeclaims, serviceaccounts and get/list/watch on pods; must NOT show cluster-scoped resources
    - Second check: `kubectl get secret ci-deployer-token -n solid-stats-staging -o jsonpath='{.data.token}' | base64 -d | cut -c1-20` (prints first 20 chars of token to confirm it is non-empty)

    ## Step 4: Extract token and CA for GitHub secrets
    - Extract SA token: `kubectl get secret ci-deployer-token -n solid-stats-staging -o jsonpath='{.data.token}' | base64 -d`
    - Extract CA: `kubectl get secret ci-deployer-token -n solid-stats-staging -o jsonpath='{.data.ca\.crt}' | base64 -d`
    - Store token as GitHub secret K8S_TOKEN (environment: staging)
    - Store CA as GitHub secret K8S_CA_CERT (environment: staging)
    - NEVER commit these values to git

    ## Step 5: Patch k3s API server certificate SAN (if 10.8.0.1 not already in SANs)
    - Verify: `openssl s_client -connect 10.8.0.1:6443 </dev/null 2>/dev/null | openssl x509 -noout -text | grep -A1 'Subject Alternative Name'` — must show `IP Address:10.8.0.1`
    - If missing: Add `tls-san: ["10.8.0.1"]` to /etc/rancher/k3s/config.yaml (or create if absent), then `sudo rm -f /var/lib/rancher/k3s/server/tls/server-ca.crt /var/lib/rancher/k3s/server/tls/server-ca.key /var/lib/rancher/k3s/server/tls/serving-kube-apiserver.crt /var/lib/rancher/k3s/server/tls/serving-kube-apiserver.key` and `sudo systemctl restart k3s`
    - Re-verify SAN after restart
    - NEVER use --insecure-skip-tls-verify in kubectl commands; if cert validation fails, fix the SAN

    ## Step 6: Configure WireGuard secrets in GitHub
    - Store WireGuard private key (CI runner peer) as WG_PRIVATE_KEY (environment: staging)
    - Store VPS WireGuard public key as WG_PEER_PUBLIC_KEY (environment: staging)
    - Store VPS endpoint (HOST:51820) as WG_ENDPOINT (environment: staging)
    - The CI runner WireGuard peer must be added to the VPS wg0 config with AllowedIPs 10.8.0.2/32

    ## Verification
    - Run the PR path (open a draft PR against master) — the validate + dry-run jobs must pass
    - `kubectl auth whoami` output in the dry-run job must not contain "system:anonymous"

    ## Troubleshooting
    - "x509: certificate is valid for X, not 10.8.0.1": SAN not patched — repeat Step 5
    - "Forbidden: namespaces is forbidden": CI tried to create namespace — check workflow YAML for any `kubectl create namespace` step
    - "secret has no token field" / empty token: k8s control plane has not yet populated the token — wait 5 seconds and retry
    - "WireGuard handshake did not complete": 51820/udp egress blocked — check VPS firewall and GitHub runner network

    See also: docs/sa-token-rotation.md for rotation cadence.
