the apply mechanism — all without touching a live S3 endpoint.

incomplete multipart uploads) are fully satisfied offline. The lifecycle JSON is the authoritative

<task type="auto">
  <name>Task 2: apply-s3-lifecycle.sh — operator apply script (S3-01)</name>
  <files>scripts/apply-s3-lifecycle.sh</files>
  <action>
  Create scripts/apply-s3-lifecycle.sh following the style of scripts/backup-postgres-now.sh.
  The script creates a one-shot k8s Job that applies the lifecycle config from inside the cluster
  so S3 credentials never leave the cluster. The lifecycle JSON is delivered to the Job via a
  temporary ConfigMap (NOT shell-variable embedding — JSON contains double-quotes which break
  any heredoc/variable substitution in shell YAML generation).

  Script structure:
  - #!/usr/bin/env bash, set -euo pipefail
  - required() helper: prints "FATAL: $1" to stderr and exits 64 if env var is unset/empty.
  - Variables: namespace="${K8S_NAMESPACE:-solid-stats-staging}",
    job_name="apply-s3-lifecycle-$(date -u +%Y%m%d%H%M%S)",
    configmap_name="s3-lifecycle-policy-$(date -u +%Y%m%d%H%M%S)",
    timeout="${JOB_TIMEOUT:-120s}",
    SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)",
    LIFECYCLE_JSON="${SCRIPT_DIR}/../config/s3/backups-lifecycle.json".
  - Abort if config/s3/backups-lifecycle.json not found:
    test -f "$LIFECYCLE_JSON" || { echo "FATAL: $LIFECYCLE_JSON not found" >&2; exit 64; }

  ConfigMap creation (avoids shell-escaping hazard):
    kubectl create configmap "$configmap_name" \
      --from-file=backups-lifecycle.json="$LIFECYCLE_JSON" \
      --namespace="$namespace" \
      -o yaml --dry-run=client | kubectl apply -f -

  Job spec embedded as a bash heredoc passed to kubectl apply -f -. The heredoc is
  safe because it uses static YAML with no embedded JSON content — only the ConfigMap
  name variable (a simple timestamp string, no special chars):
    Job name: $job_name, namespace: $namespace.
    Container image: postgres:17-alpine + apk add --no-cache aws-cli (mirrors backup pattern;
    no new image dependency; no legitimacy audit needed).
    Volumes: one volume named "lifecycle-config" of type configMap, configMapName: $configmap_name.
    volumeMounts: mountPath /config, name lifecycle-config, readOnly true.
    Mount server-2-runtime as env vars: AWS_ACCESS_KEY_ID from S3_ACCESS_KEY_ID key,
    AWS_SECRET_ACCESS_KEY from S3_SECRET_ACCESS_KEY key, S3_BUCKET from S3_BUCKET key.
    Env literals: AWS_DEFAULT_REGION=ru-1, AWS_EC2_METADATA_DISABLED=true,
    S3_ENDPOINT=https://s3.twcstorage.ru.
    Container command (sh -ec):
      1. apk add --no-cache aws-cli
         export AWS_CONFIG_FILE=/tmp/aws-config
         aws configure set default.s3.addressing_style path
      2. GET-before-PUT:
         get_output=$(aws --endpoint-url="$S3_ENDPOINT" s3api get-bucket-lifecycle-configuration \
           --bucket "$S3_BUCKET" 2>&1) && get_rc=0 || get_rc=$?
         if echo "$get_output" | grep -q "NotImplemented"; then
           echo "FATAL: Timeweb S3 endpoint does not support lifecycle API. S3-03 empirical proof did not complete successfully. Do not apply retention policy." >&2
           exit 1
         elif [ "$get_rc" -eq 0 ]; then
           echo "WARN: bucket already has a lifecycle configuration — review before applying" >&2
         elif echo "$get_output" | grep -q "NoSuchLifecycleConfiguration"; then
           echo "INFO: no existing lifecycle config (expected on first apply)"
         else
           echo "FATAL: unexpected error from get-bucket-lifecycle-configuration: $get_output" >&2
           exit 1
         fi
      3. aws --endpoint-url="$S3_ENDPOINT" s3api put-bucket-lifecycle-configuration \
           --bucket "$S3_BUCKET" \
           --lifecycle-configuration file:///config/backups-lifecycle.json
      4. echo "lifecycle configuration applied successfully"
    Job spec: restartPolicy Never, backoffLimit 0, ttlSecondsAfterFinished 86400.
    ServiceAccount: use postgres-backup SA (already exists, no S3 RBAC — correct; SA is for pod
    identity only, not k8s API access). Set automountServiceAccountToken: false.
    securityContext: allowPrivilegeEscalation: false.
    Resources: requests cpu 50m memory 128Mi; limits cpu 200m memory 256Mi.

  After Job creation:
  - kubectl -n "$namespace" wait --for=condition=complete "job/$job_name" --timeout="$timeout";
    on non-zero exit: print logs and exit 1.
  - Print logs on success: kubectl -n "$namespace" logs "job/$job_name".
  - Delete Job: kubectl -n "$namespace" delete "job/$job_name" --ignore-not-found
    (ttl also covers cleanup).
  - Delete ConfigMap: kubectl -n "$namespace" delete configmap "$configmap_name" --ignore-not-found.

  Comment at top of script: "NOTE: The Job spec is created inline (kubectl apply -f -); no manifest
  file is written to k8s/staging/. This Job is purely operator-driven and never reaches the CD glob."
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator machine → k8s API | apply-s3-lifecycle.sh submits a Job via kubectl; kubectl creds from local kubeconfig or KUBECONFIG env var |
| k8s Job → S3 endpoint | Job pod holds S3 creds from server-2-runtime Secret; creds never written to disk or echoed in logs |
| config/s3/ directory → git | lifecycle JSON contains no secrets — only prefix strings and day counts; safe to commit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | config/s3/backups-lifecycle.json | mitigate | validate-s3-lifecycle.py asserts Prefix == "backups/postgres/" and Days >= 30 in CI; drift from mis-edit is caught before operator runs apply |
| T-10-02 | Tampering | apply-s3-lifecycle.sh overwrites existing lifecycle config | mitigate | GET-before-PUT: script calls get-bucket-lifecycle-configuration first; prints WARN if existing config found; operator must review before applying |
| T-10-03 | Information Disclosure | S3 creds (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) | mitigate | creds stay in-cluster (k8s Secret → Job env vars); script never echoes secret values; aws-cli does not log them by default |
| T-10-04 | Denial of Service | Expiration rule too broad (no prefix filter) | mitigate | Rule 1 Filter.Prefix = "backups/postgres/" scopes deletion; Rule 2 AbortIncompleteMultipartUpload is bucket-wide but targets only in-progress uploads, not completed objects |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | no package manager used; aws-cli installed via apk inside the Job container at runtime, matching the existing backup pattern; no new registry packages in this plan |
</threat_model>
